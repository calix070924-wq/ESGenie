# OCR 표 복원 캐스케이드 & 검증 게이트 설계

작성일: 2026-06-22
대상: ESGenie OCR 파이프라인 (정형=Azure Document Intelligence, 비정형=gpt-4.1-mini)

---

## 1. 설계 원칙

1. **성능 우선, 속도는 나중.** 정확도 천장을 먼저 찍고, 그 다음에 정확도를 거의 안 깎으면서 속도를 최적화한다. 지금 단계에서 비용·레이턴시 걱정은 의도적으로 보류한다.
2. **캐스케이드(사다리) 구조.** 쉬운 처리부터 어려운 처리까지 티어를 나누고, 위에서부터 차례로 시도한다. 통과하면 멈추고, 실패하면 다음 티어로 내려간다.
3. **검증 게이트가 사다리의 심장.** "됐냐 / 안 됐냐"를 자동으로 판정하는 게이트가 없으면 캐스케이드는 작동하지 않는다. 게이트가 다음 티어로 내려보낼지를 결정한다.
4. **실패는 안전한 쪽으로 기운다.** 애매하면 더 센(비싼) 처리로 보낸다. 최후에는 "사람 확인 필요" 플래그를 단다 — 틀린 값을 조용히 흘려보내지 않는다.
5. **추적 가능성.** 모든 값은 "어느 페이지, 어느 좌표, 어느 티어에서 나왔는지"를 메타데이터로 달고 다닌다. 나중에 실사 증빙으로 역추적해야 한다.

## 2. 전체 파이프라인 개요

```
문서 입력
  │
  ▼
[분기 판단]  정형 / 비정형  ← LLM 프롬프트 자동 분류 (애매하면 비정형으로)
  │
  ▼
┌─────────────────────────────────────────────┐
│  표 복원 캐스케이드                          │
│                                             │
│  Tier 0  Azure 기본 표 추출 ──▶ [게이트] ─┐ │
│  Tier 1  규칙 기반 복원     ──▶ [게이트] ─┤ │
│  Tier 2  LLM 구조 재구성    ──▶ [게이트] ─┤ │
│  Tier 3  비전 모델(이미지)  ──▶ [게이트] ─┤ │
│  Tier 4  사람 확인 플래그                  │ │
│                                          통과│
└──────────────────────────────────────────┼─┘
                                           ▼
                                    구조화된 표 + 메타데이터 → RAG/보고서
```

각 티어 끝에 **검증 게이트**가 붙는다. 게이트 결과는 셋 중 하나:
- **ACCEPT** — 충분히 좋다. 멈추고 결과 채택.
- **ESCALATE** — 부족하다. 다음 티어로 내려보냄.
- **HUMAN** — 자동으로는 한계. 사람 확인 플래그.

## 3. 검증 게이트 설계 (핵심)

### 3.1 게이트의 본질 — "정답"이 아니라 "정답일 확률"을 잰다

우리는 정답표(ground truth)가 없다. 그래서 게이트는 정확도를 **직접** 재는 게 아니라, "이 추출이 틀렸을 법한 징후"를 잡아내는 **프록시(대리 지표)** 들의 모음이다. 징후가 적으면 통과, 많으면 내려보낸다.

게이트는 4종류의 신호를 본다:

| 신호 | 무엇을 보나 | 비용 |
|------|------------|------|
| A. 구조 정합성 | 표 모양이 말이 되는가 | 매우 쌈 (규칙) |
| B. 수치 정합성 | 숫자들이 서로 맞는가 | 쌈 (규칙) |
| C. 추출기 신뢰도 | 추출기 자신이 얼마나 확신하나 | 공짜 (Azure가 제공) |
| D. 의미 검증 | 원본이랑 진짜 일치하나 | 비쌈 (LLM/비전) |

쌀수록 먼저 돌린다. A·B·C로 대부분 걸러내고, D는 정말 애매할 때만 쓴다.

### 3.2 신호 A — 구조 정합성 (규칙 기반, 항상 실행)

- **A1. 직사각형성(rectangularity).** 모든 행의 열 개수가 같은가? 행마다 칸 수가 들쭉날쭉하면(ragged) 셀 병합/누락을 잘못 푼 것. → 불일치 행 비율 **> 5%** 면 의심.
- **A2. 빈 칸 비율.** 전체 셀 중 빈 칸 비율. ESG 표는 보통 촘촘하다. → **> 20%** 면 행/열 누락 의심.
- **A3. 헤더 존재.** 첫 1~2행이 숫자가 아닌 라벨(항목명)로 채워졌는가? 헤더가 숫자면 헤더-데이터 경계를 잘못 잡은 것.
- **A4. 열 타입 일관성.** 한 열의 80% 이상이 숫자면 그 열은 "수치 열". 그 안에 섞인 텍스트 셀은 OCR 오인식 후보(예: `1,234`를 `l,234`로). → 수치 열의 비(非)숫자 셀 비율 **> 10%** 면 의심.

### 3.3 신호 B — 수치 정합성 (규칙 기반, ESG 특화, 강력)

이게 ESG 데이터에서 제일 강력한 게이트다. 숫자는 거짓말을 못 한다.

- **B1. 합계 검증.** "합계 / 소계 / Total" 행·열이 있으면, 구성 요소의 합과 같은가? 반올림 오차 허용:
  - 상대 오차 **≤ 0.5%** 또는 절대 오차 **≤ 1단위** 중 큰 쪽 이내면 통과.
  - 어긋나면 숫자가 어긋났거나(자리 밀림) 행이 빠진 것 → 강한 ESCALATE 신호.
- **B2. 단위 존재.** 수치 열에 단위(tCO2eq, %, MWh 등)가 헤더나 인접 셀에 있는가? 단위 없는 숫자는 보고서에서 무의미. (search_terms 보강 때 unit 필드 추가한 것과 연결)
- **B3. 자릿수/범위 상식.** 같은 열 안에서 자릿수가 갑자기 튀는 값(예: 다 4자리인데 혼자 7자리)은 천단위 콤마 오인식 후보. → 플래그만, 단독으로 ESCALATE는 아님.
- **B4. 연도/기간 단조성.** 시계열 표면 연도 헤더가 순서대로(2022→2023→2024) 가는가? 깨지면 열 순서 뒤섞임.

### 3.4 신호 C — 추출기 신뢰도 (공짜, Azure 한정)

Azure Document Intelligence는 셀마다 confidence 점수를 준다. 공짜로 나오니 무조건 쓴다.

- **C1. 평균 신뢰도.** 표 전체 셀 평균 confidence. → **< 0.85** 면 의심.
- **C2. 저신뢰 셀 비율.** confidence **< 0.6** 인 셀의 비율. → **> 10%** 면 ESCALATE. (저신뢰 셀이 **수치 열**에 있으면 가중치 2배 — 숫자 오인식이 제일 치명적)

### 3.5 신호 D — 의미 검증 (비쌈, 최후 직전에만)

규칙·신뢰도로 판정이 안 서는 회색지대에서만 호출한다.

- **D1. LLM 자기검증.** 추출한 표(마크다운)와 원본 표 텍스트를 같이 주고 "행/열/숫자가 일치하나? 빠지거나 어긋난 곳을 짚어라"고 시킨다.
- **D2. 비전 대조.** 표 영역을 이미지로 크롭해서 비전 모델에 원본 그대로 읽히고, 추출 결과와 셀 단위로 대조한다. (Tier 3에서 이미 비전을 쓰면 그 출력 자체가 D2 역할)

### 3.6 게이트 판정 정책 (신호 → 결정)

각 신호는 점수가 아니라 **위반 플래그**로 환산한다. 그리고 다음 규칙으로 결정:

```
강한 위반 (hard fail) — 하나라도 걸리면 즉시 ESCALATE:
  - B1 합계 불일치
  - C2 저신뢰 셀 비율 > 10%
  - A1 비직사각형 행 비율 > 5%

약한 위반 (soft flag) — 점수로 합산:
  - A2, A3, A4, B2, B3, B4, C1  → 각 1점

결정:
  - hard fail 0개  AND  soft flag ≤ 1점  → ACCEPT
  - hard fail 0개  AND  soft flag 2~3점  → 회색지대: 신호 D 호출 → D 통과 시 ACCEPT, 아니면 ESCALATE
  - hard fail ≥ 1개                       → ESCALATE
  - 마지막 티어(Tier 3)에서도 ESCALATE 판정 → HUMAN
```

> 임계치(5%, 20%, 0.85, 0.6, 10%, 0.5% 등)는 **초기 추정값**이다. 라벨링한 표 50~100개로 보정(ROC 보고 조정)하는 것을 전제로 한다. 4절 마지막 참고.

## 4. 티어별 상세

### Tier 0 — Azure 기본 표 추출
- 하는 일: Azure Document Intelligence로 표 구조 + 셀 confidence 추출.
- 통과 게이트: 3.6 정책 전체.
- 실패 시 → Tier 1.

### Tier 1 — 규칙 기반 복원 (결정적, 싸고 안전)
- 하는 일: LLM 없이 코드로 고칠 수 있는 것만 고친다.
  - 병합셀 값 전파(merged-cell fill-down/right)
  - 다단 헤더 평탄화(2줄 헤더 → "대분류_소분류" 결합)
  - 수치 열 정규화(콤마/공백 제거, `O→0`, `l→1` 같은 안전한 치환)
  - 명백한 빈 행/열 제거
- 통과 게이트: 3.6 재실행. 특히 B1 합계가 이제 맞는지 다시 본다.
- 실패 시 → Tier 2.

### Tier 2 — LLM 구조 재구성
- 하는 일: gpt-4.1-mini에게 Tier 0/1의 텍스트를 주고 표 구조를 다시 짜게 한다. "헤더가 몇 줄인지, 어떤 셀이 병합인지" 추론시킴.
- 주의: LLM이 숫자를 **바꾸지 못하게** 한다 — 구조만 재배치, 값은 원본 보존(프롬프트로 강제 + 사후 대조).
- 통과 게이트: 3.6 재실행 + D1(자기검증) 가중.
- 실패 시 → Tier 3.

### Tier 3 — 비전 모델 (이미지 통째로)
- 하는 일: 표 영역을 이미지로 크롭해 비전 모델에 원본 그대로 읽힌다. 텍스트 레이어가 깨진 스캔본·복잡한 레이아웃의 최후 자동 수단.
- 통과 게이트: 3.6 재실행. 여기까지도 ESCALATE면 → Tier 4.

### Tier 4 — 사람 확인 플래그
- 하는 일: 자동 채택하지 않는다. "검토 필요" 큐에 넣고, 어느 게이트가 왜 걸렸는지(예: "합계 불일치, 12페이지 배출량 표") 사유를 함께 단다.
- 핵심: **틀린 값을 맞는 척 통과시키지 않는 것**이 캐스케이드의 최종 안전장치.

## 5. 메타데이터 & 추적

추출된 모든 셀/값에 다음을 붙여 저장한다:

- `source_page` — 원본 페이지 번호
- `bbox` — 셀의 좌표(역추적·증빙 하이라이트용)
- `resolved_tier` — 어느 티어에서 최종 채택됐는지 (0~3)
- `confidence` — 게이트 통과 시점의 종합 신뢰 신호
- `gate_flags` — 통과 시 남아 있던 약한 위반들 (사후 감사용)

이 정보가 있어야 "이 배출량 숫자는 보고서 12페이지 표에서 Tier 2로 복원됨, 합계검증 통과"처럼 실사 증빙으로 연결된다. (공유 증빙 모듈과 직결)

## 6. 미해결 / 현업(개발자)에게 확인할 것

1. **Azure가 주는 confidence를 지금 파이프라인에서 실제로 받아 쓰고 있나?** (신호 C가 공짜인지, 버려지고 있는지 확인)
2. **분기 분류기(정형/비정형)의 오분류율을 측정하고 있나?** 애매한 케이스를 비정형으로 보내는 안전 기울임이 들어가 있나?
3. **Tier 1 규칙 복원이 지금 코드에 일부라도 있나, 아니면 Azure 출력을 그대로 RAG에 넣고 있나?**
4. **표 영역 좌표(bbox)와 페이지 번호를 지금 저장하고 있나?** (5절 추적이 가능한 상태인지)
5. **라벨링된 표 평가셋이 있나?** 게이트 임계치를 보정하려면 "정답 표 50~100개"가 필요하다. 없으면 이게 다음 1순위 작업.
6. **합계 검증(B1)을 걸 수 있을 만큼, 표 안에 'Total/합계' 행이 실제로 자주 있나?** ESG 표 분포를 알아야 B1의 적용 범위가 정해진다.

## 7. 구현 우선순위 (제안)

1. **PR-A: 게이트 골격 + 신호 A·B·C 구현.** Tier 0 뒤에 게이트를 붙이고, 규칙 기반 신호부터. (D 없이도 대부분 걸러짐)
2. **PR-B: Tier 1 규칙 복원.** 가장 가성비 높은 정확도 상승. LLM 안 쓰고 합계 맞추기.
3. **PR-C: 평가셋 구축 + 임계치 보정.** 라벨 50~100개로 6절 임계치 튜닝.
4. **PR-D: Tier 2/3 LLM·비전 캐스케이드 + 신호 D.**
5. **PR-E: 메타데이터·추적 + Tier 4 휴먼 큐.**

> 속도 최적화(캐싱, top-k 축소, 티어 스킵 휴리스틱)는 정확도가 안정된 PR-C 이후에 별도 트랙으로.
