# ESGenie Streamlit UI 리디자인 설계

## 목적

현재 ESGenie Streamlit UI는 기능은 많지만 입력, 분석, 결과, 다운로드가 한 화면 계층에 섞여 있어 첫 인상이 복잡하고 작업 흐름이 끊긴다.  
이번 리디자인의 목적은 단순히 "예쁘게" 만드는 것이 아니라 아래 세 가지를 동시에 달성하는 것이다.

1. 사용자가 현재 상태를 즉시 이해할 수 있게 한다.
2. 입력 → 분석 → 검토 → 산출물 다운로드 흐름을 자연스럽게 만든다.
3. 스타일과 레이아웃 책임을 `app.py`에서 분리해 이후 기능 추가 비용을 낮춘다.


## 현재 UI 문제 정의

기준 파일:

- `app.py`
- `esgenie/ui/tabs.py`

핵심 문제는 아래와 같다.

1. 입력 UI가 분산되어 있다.
   사이드바에는 회사/업종/연도/분석 버튼이 있고, 본문에는 파일 업로드와 정성 설문이 있다. 사용자는 "분석 준비가 끝났는지"를 한 번에 판단하기 어렵다.

2. 결과 탭이 너무 많고 동급으로 나열되어 있다.
   홈, SSOT, 공시 진단, 초안, 검증, 규정 검증, 감사 추적, 실사 응답서, 벤치마크가 모두 1차 네비게이션에 있다. 중요도가 낮은 기능이 핵심 흐름과 경쟁한다.

3. 정보 위계가 없다.
   KPI, 설명문, 긴 표, 원문, 다운로드 버튼이 같은 시각적 무게로 등장한다. "지금 무엇을 먼저 봐야 하는지"가 드러나지 않는다.

4. 스타일이 인라인 CSS와 개별 `st.markdown`에 퍼져 있다.
   테마 변경이나 공통 컴포넌트 재사용이 어렵다.

5. 제품 성격이 명확히 드러나지 않는다.
   현재는 데모 느낌과 운영형 대시보드 느낌이 혼재한다. ESGenie는 "ESG 분석 워크벤치" 쪽이 더 적합하다.


## 디자인 방향

### 제품 톤

- 방향: 엔터프라이즈 분석 워크벤치
- 인상: 차분하고 신뢰감 있는 운영 화면
- 포인트: "복잡한 분석을 정리해서 보여준다"는 느낌

### 시각 원칙

- 메인 컬러는 선명한 초록 1색이 아니라 `deep green + slate + warm neutral` 조합으로 간다.
- 상태 표현은 색보다 구조를 우선한다.
- 카드, 패널, 배지, 표 상단 요약을 공통 컴포넌트로 통일한다.
- 긴 텍스트는 카드형 컨테이너로 읽기 영역을 분리한다.
- 개발자용 기능은 전면에 두지 않고 실험실 성격으로 약하게 노출한다.

### 제안 컬러 토큰

- `--bg-app`: `#F4F6F2`
- `--bg-panel`: `#FBFCFA`
- `--bg-elevated`: `#FFFFFF`
- `--text-strong`: `#162218`

- `--accent-secondary`: `#89A88E`
- `--status-success`: `#2E7D32`

- `--status-danger`: `#B7463B`

## 새 정보 구조

기존 9개 탭을 아래 5개 영역으로 재구성한다.

1. `Overview`
   현재 분석 상태, 핵심 KPI, 위험도, 다음 액션, 최종 요약

2. `Analysis`
   공시 진단, 초안, 검증을 한 흐름으로 묶은 핵심 분석 화면

3. `Evidence`
   업로드 파일, SSOT, provenance, 규정 검증

4. `Deliverables`
   최종 보고서, 공급망 실사 응답서, 다운로드 묶음

5. `Lab`
   벤치마크, 디버그성 기능, 내부 실험용 기능

### 재매핑

- `홈` → `Overview`
- `공시 진단`, `보고서 생성`, `검증 & 최종본` → `Analysis`
- `증빙 & SSOT`, `규정 검증`, `감사 추적 & HITL` → `Evidence`
- `실사 응답서` + 다운로드 액션 → `Deliverables`
- `벤치마크` → `Lab`


## 메인 레이아웃

페이지는 아래 4단 구조로 고정한다.

1. Global Header
2. Analysis Setup Panel
3. Summary Strip
4. Main Navigation Content

### 1. Global Header

항상 최상단에 표시한다.

- 회사명
- 보고 연도
- 프로파일
- 분석 영역
- 현재 결과 상태 배지
- 마지막 실행 기준 정보
- `분석 시작` 또는 `다시 분석` 버튼

목적:

- 사용자가 현재 어떤 대상을 보고 있는지 즉시 파악
- stale result 여부를 명확히 표시

### 2. Analysis Setup Panel

입력 요소를 한 컨테이너에 모은다.

- 회사 검색 / 직접 입력
- DART 코드 / 연동 여부
- 업종 / 연도 / 분석 영역
- 고급 설정
- 증빙 업로드
- 설문 입력 요약

패널 내부는 3개 블록으로 쪼갠다.

- `Company Context`
- `Evidence Intake`
- `Advanced Controls`

설문 전체를 바로 펼치지 말고:

- 기본 상태에서는 "입력된 항목 수"만 보여준다.
- `정성 항목 입력` 버튼 또는 expander로 진입한다.

### 3. Summary Strip

분석이 끝난 뒤 헤더 바로 아래에 핵심 KPI를 카드로 표시한다.

- 커버리지
- SSOT 노드 수
- 증빙 확인률
- 규정 통과율
- 최종 위험도
- 검토 필요 건수

원칙:

- 숫자만 두지 않고 짧은 상태 문장도 같이 둔다.
- 위험도 카드만 강조 색상과 배지를 가진다.

### 4. Main Navigation Content

탭 또는 segmented control을 사용한다.

권장 순서:

- `Overview`
- `Analysis`
- `Evidence`
- `Deliverables`
- `Lab`

## 페이지별 상세 설계

### Overview

역할:

- 첫 화면에서 "지금 결과가 어떤 상태인지"만 빠르게 전달한다.

구성:

- 상태 요약 카드
- 다음 액션 패널
- 최종 보고서 요약
- 검토 필요 알림

제거할 것:

- 긴 상세 테이블
- 너무 많은 원문 노출

### Analysis

역할:

- 공시 진단 → 초안 → 검증을 하나의 분석 플로우로 보여준다.

구성 순서:

1. 공시 진단 상단 요약
2. ISSB/KSSB 갭과 선택적 공시 탐지
3. 초안/최종본 비교 카드
4. 위험도 추이 그래프
5. 4축 리스크 분해
6. 세부 공시/리스크 표

핵심 개선:

- 지금은 초안과 검증이 탭으로 분리되어 맥락이 끊긴다.
- 비교 뷰로 묶으면 재생성의 효과가 즉시 드러난다.

### Evidence

역할:

- 어떤 증빙이 어떤 결과를 만들었는지 추적한다.

구성 순서:

1. 업로드 상태 요약
2. SSOT 요약 지표
3. 증빙 노드/정성 노드 테이블
4. 규정 검증 결과
5. provenance 및 HITL 상세

핵심 개선:

- 현재는 증빙, 규정, 감사 추적이 따로 놀고 있다.
- 모두 "결과의 근거"라는 하나의 축으로 재정렬한다.

### Deliverables

역할:

- 사용자가 실제로 가져가는 결과물을 정리한다.

구성 순서:

1. 보고서 다운로드
2. 실사 응답서 다운로드
3. 응답서 커버리지/플래그 요약
4. 문항별 상세 근거

핵심 개선:

- 다운로드 버튼을 Evidence 탭 안쪽에 숨기지 않는다.
- 산출물 중심의 마감 화면을 만든다.

### Lab

역할:

- 운영 사용자의 기본 흐름에서 개발/실험 기능을 분리한다.

구성:

- 그린워싱 벤치마크
- 향후 디버그 패널


## 컴포넌트 설계

공통 UI를 함수 단위로 분리한다.

### 필수 컴포넌트

- `page_header(...)`
  페이지 제목, 설명, 상태 배지, 우측 액션

- `metric_card(...)`
  숫자 + 상태 설명 + 델타

- `status_chip(...)`
  success, warning, danger, neutral 배지

- `section_panel(...)`
  연한 배경의 구획 카드

- `empty_state(...)`
  아직 분석하지 않은 화면 공통 처리

- `report_card(...)`
  초안/최종본 텍스트 렌더링 공통 카드

- `action_callout(...)`
  다음 행동 제안 박스

- `dataframe_section(...)`
  표 제목, 요약 설명, 필터 영역, 데이터프레임 래퍼

### 선택 컴포넌트

- `risk_summary_banner(...)`
- `upload_summary_panel(...)`
- `download_tile(...)`
- `stale_result_notice(...)`

## 상태 관리 원칙

리디자인하면서 아래 상태를 명시적으로 구분한다.

- `current_inputs`
- `last_run_inputs`
- `result`
- `is_result_stale`

필요 이유:

- 사용자가 입력을 바꾼 뒤 아직 재분석하지 않은 상태를 구분해야 한다.
- 현재 결과가 어떤 설정 기준으로 생성됐는지 보여줄 수 있어야 한다.

예시 UX:

- 업종이나 영역을 바꾸면 헤더에 `설정 변경됨 - 결과 갱신 필요` 배지 표시
- `다시 분석` 버튼 강조


## 구현 우선순위

### Phase 1. 기반 정리

- `theme.py`, `components.py` 생성
- `app.py`에서 인라인 CSS 제거
- 글로벌 헤더와 summary strip 도입

### Phase 2. 정보구조 개편

- 탭 9개를 5개로 재구성
- `Overview`, `Analysis`, `Deliverables` 우선 적용

### Phase 3. 근거 화면 정리

- `Evidence`에 SSOT/규정/감사 추적 통합
- 업로드 상태 요약 패널 추가

### Phase 4. 마감 품질

- 테이블 상단 설명/필터/상태 배지 정리
- plotly 차트 색상과 여백 통일
- empty state, warning state 통일


## 구현 시 주의점

1. Streamlit 기본 컴포넌트 한계를 전제로 설계한다.
   너무 많은 커스텀 HTML 인터랙션에 기대지 않는다.

2. 접근성보다 화려함을 우선하지 않는다.
   상태 색상은 텍스트 라벨과 함께 제공한다.

3. 현재 테스트 자산과 충돌하지 않도록 렌더 함수명 변경 범위를 관리한다.

4. 다운로드와 분석 실행 흐름은 기능 회귀가 나기 쉬우므로 마지막에 한 번 더 확인한다.


## 첫 구현 작업 제안

첫 PR 또는 첫 작업 단위는 아래 범위가 적절하다.

1. `theme.py`와 `components.py` 추가
2. `app.py`의 헤더/입력 구역 재배치
3. 기존 `홈` 탭을 `Overview` 화면으로 교체
4. 기존 `공시 진단 + 초안 + 검증`을 `Analysis`로 합치는 골격만 우선 반영

이 단계까지만 해도 체감 품질이 크게 올라가고, 이후 나머지 화면 정리는 같은 패턴으로 이어갈 수 있다.
