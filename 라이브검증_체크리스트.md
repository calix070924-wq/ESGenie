# ESGenie 라이브 파이프라인 검증 체크리스트

> 목적: **현재 Azure 스택(gpt-4.1-mini judge + Azure Doc Intelligence + sbert 임베딩)으로
> L0~L5 풀 파이프라인이 mock 폴백 없이 라이브로 1회 완주**하는지 증명한다.
> 대회 전 1순위 리스크. 반드시 **네 로컬 PC**에서 실행 (Claude 샌드박스는 외부망 차단).


## 1. 핵심 관문 — STRICT 모드 풀 파이프라인 (judge 라이브)

`ESGENIE_STRICT=1`이면 **API 실패 시 조용히 mock으로 안 빠지고 예외를 던진다.**
즉 이게 통과하면 = 진짜 라이브로 돈 것.

```bash
ESGENIE_STRICT=1 python -m esgenie.pipeline --ticker 005930 --areas E --llm-judge
```

- `--llm-judge` 빼면 judge LLM이 안 붙어서 검증 의미 없음 → **반드시 포함**
- `LLMUnavailableError`가 뜨면 → 라이브 실패 (키/엔드포인트/네트워크 문제). 여기서 멈추고 원인부터 해결
- 끝까지 완주하고 위험도가 출력되면 → judge가 실모델로 돈 것

## 2. 증거 확인 — trace에 mock이 없어야 함

방금 생성된 audit_trace의 `model_versions`를 직접 까서 확인:

```bash
latest=$(ls -t outputs/audit_trace_005930_E_*.json | head -1)
python - "$latest" <<'PY'
import json, sys
d = json.load(open(sys.argv[1]))
mv = d["sentences"][0]["model_versions"]
print("llm          :", mv["llm"])            # ✅ gpt-4.1-mini  ❌ mock
print("embed_backend:", mv["embed_backend"])  # ✅ sbert         ❌ hash-fallback
PY
```

**합격 기준**
- `llm = gpt-4.1-mini` (또는 실제 모델명) — **`mock` 아니어야 함**
- `embed_backend = sbert` — **`hash-fallback` 아니어야 함**

## 3. OCR 채널 라이브 확인 (증빙 업로드 경로)

OCR은 메인 파이프라인과 별개 경로(`ocr_router`). Upstage 실패 시 디지털PDF→mock으로
**조용히 폴백**되므로, engine 값을 직접 봐야 진짜 Upstage로 돈 건지 안다.
한 줄 검증: `python -m scripts.verify_upstage_live`

```bash
python - <<'PY'
from esgenie.config import SETTINGS  # .env 로드(load_dotenv) 먼저
from esgenie.ssot import ocr_router
for f in ["data/test_docs/kepco_bill_2025_12.pdf",
          "data/test_docs/waste_ledger_2025.pdf",
          "data/test_docs/safety_policy_2025.pdf"]:
    dec = ocr_router.route_document(f)
    ext = ocr_router.extract_document(f, dec)
    m = getattr(ext, "router_meta", {})
    print(f"{f.split('/')[-1]:28} engine={m.get('engine')} upstage_error={str(m.get('upstage_error'))[:50]}")
PY
```

**합격 기준**
- 정형(kepco/waste) → `engine=upstage_dp`, `upstage_error=None`
- 비정형(safety_policy) → `engine=gpt-4.1-mini-text`
- `engine=None` 이나 `mock`이 뜨면 → Upstage 호출 실패하고 폴백 중 (원인 추적 필요)

## 자주 터지는 지점 & 대응

| 증상 | 원인 | 대응 |
|---|---|---|
| `LLMUnavailableError` | Azure OpenAI 키/엔드포인트/배포명 불일치 | `.env`의 `AZURE_OPENAI_ENDPOINT`·배포(deployment)명 확인 |
| `embed_backend=hash-fallback` | sentence-transformers 미설치 or 모델 다운로드 실패 | `pip install sentence-transformers`, 첫 실행 시 모델 다운로드 네트워크 필요 |
| OCR `engine=None` | route_document가 ambiguous로 분류 | 문서 타입 라우팅 로직 점검 (정형이 ambiguous로 새는지) |
| `engine` 가려짐 | router_meta 덮어쓰기 (과거 버그) | 6/13 수정분 반영됐는지 확인 |
| 천 단위 숫자 깨짐(`128, 400`) | OCR 콤마 뒤 공백 | `_find_number` 정규화 반영됐는지 확인 |
| OCR `engine=None`/`mock` + `upstage_error` 있음 | `UPSTAGE_API_KEY` 미설정/오타 or 망 차단 | `.env`의 `UPSTAGE_API_KEY` 확인, `scripts.verify_upstage_live` 실행 |
