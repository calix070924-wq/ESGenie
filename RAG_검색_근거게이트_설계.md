# RAG 검색 & 근거 게이트 설계

작성일: 2026-06-22
대상: ESGenie RAG 파이프라인 (KESGItem의 search_terms/unit 기반 evidence 검색 → 보고서/실사 응답 생성)
짝 문서: `OCR_표복원_검증게이트_설계.md` (동일한 "게이트 사다리" 철학)

---

## 1. 설계 원칙

OCR 문서와 같다. 한 가지가 더 추가된다.

1. **성능 우선, 속도는 나중.** 검색 정확도(재현율) 천장을 먼저 찍고, 그 다음에 정확도 거의 안 깎으며 속도 최적화.
2. **캐스케이드 구조.** 싸고 정확한 검색부터, 비싸고 강한 검색까지 티어로. 통과하면 멈춤.
3. **게이트가 사다리의 심장.** "잘 찾았냐"를 자동 판정.
4. **실패는 안전한 쪽으로.** 못 찾았으면 지어내지 말고 "근거 없음"으로 빠지거나 사람에게.
5. **추적 가능성.** 답의 모든 값은 "어느 청크 → 어느 문서 → 어느 페이지"로 역추적된다.
6. **(RAG 고유) 실패 지점이 둘이다.** ① 검색 실패: 근거를 못 찾음. ② 생성 실패: 찾았는데 무시하거나 지어냄. **게이트도 두 개**여야 한다.

## 2. 전체 파이프라인 개요

```
질의(KESGItem / RBA 항목)
  │
  ▼
[질의 구성]  search_terms·동의어·단위 확장
  │
  ▼
┌──────────────────────────────────────────────┐
│  검색 캐스케이드                              │
│  Tier 0  키워드(BM25)+search_terms ─▶[검색게이트]┐│
│  Tier 1  하이브리드(BM25+임베딩)   ─▶[검색게이트]┤│
│  Tier 2  질의확장+리랭크           ─▶[검색게이트]┤│
│  Tier 3  에이전트형 반복검색       ─▶[검색게이트]┤│
│  Tier 4  "근거 없음" / 사람 확인               ││
└────────────────────────────────────────────┬─┘
                                             통과
                                              ▼
                                      [답변 생성] 출처 강제 인용
                                              ▼
                                       ┌──[근거 게이트]──┐
                                  통과 │                 │ 실패
                                       ▼                 ▼
                              답변 + 증빙 링크      재생성 / 근거없음 / 사람
```

게이트 결정은 OCR과 동일하게 셋: **ACCEPT / ESCALATE / HUMAN**.

## 3. 게이트 A — 검색 게이트 (생성 *전*)

"지금 가져온 청크들로 답할 수 있나?"를 LLM 호출 **전에** 판정한다. 여기서 거르면 비싼 생성을 헛쓰지 않는다.

### 3.1 신호

| 신호 | 무엇을 보나 | 비용 |
|------|------------|------|
| R1. 최상위 점수 | top-1 청크가 충분히 닮았나 | 공짜 |
| R2. 점수 마진 | top-1과 나머지의 점수 격차 | 공짜 |
| R3. 필드 커버리지 | 항목이 요구하는 조각을 다 찾았나 | 쌈 |
| R4. 방식 합의 | 키워드·임베딩 결과가 겹치나 | 쌈 |

- **R1. 최상위 점수.** top-1 유사도(또는 BM25 점수 정규화값)가 절대 임계 이상인가. → **< 0.6** 이면 의심. 아무것도 안 닮았는데 1등이라고 답하면 안 됨.
- **R2. 점수 마진.** top-1과 top-5 꼬리의 점수 차. 다 고만고만하면(마진 작음) 검색기가 헷갈린 것 — 정답이 섞여 있을 확률 낮음. → 마진 **< 0.1** 이면 의심.
- **R3. 필드 커버리지 (ESG 특화, 강력).** 하나의 K-ESG/RBA 항목은 보통 **값 + 단위 + 기간 + 출처**를 다 요구한다. 가져온 청크들이 이 네 조각을 다 덮는가?
  - 단위가 필요한 항목인데(unit 필드 존재) 어떤 청크에도 단위가 없다 → 강한 ESCALATE.
  - search_terms 중 핵심어가 어떤 청크에도 안 잡힘 → ESCALATE.
- **R4. 방식 합의.** BM25 top-k와 임베딩 top-k의 교집합 비율. 두 방식이 같은 청크를 가리키면 신뢰 ↑. → 겹침 **< 20%** 면 의심(둘이 딴소리 = 불안정).

### 3.2 판정 정책

```
hard fail (하나라도 → ESCALATE):
  - R1 top-1 < 0.6
  - R3 필수 단위/핵심어 미커버

soft flag (각 1점):
  - R2 마진 작음, R4 합의 낮음, R3 일부 필드 누락

결정:
  - hard 0 AND soft ≤ 1  → ACCEPT (생성으로)
  - hard 0 AND soft 2~3  → 다음 티어로 ESCALATE (회색지대는 더 센 검색이 싸게 먹힘)
  - hard ≥ 1             → ESCALATE
  - Tier 3에서도 실패     → "근거 없음" 답변 + HUMAN 플래그
```

## 4. 검색 캐스케이드 (티어)

### Tier 0 — 키워드(BM25) + search_terms
- 정확한 용어가 문서에 그대로 있으면 제일 싸고 정확. ESGReveal식 search_terms/동의어로 질의 확장("온실가스/GHG/탄소배출").
- 실패 → Tier 1.

### Tier 1 — 하이브리드 (BM25 + 임베딩)
- 키워드로 안 잡히는 의미적 표현("기후 관련 배출")을 임베딩이 잡음. 두 점수를 결합(RRF 등).
- 대부분의 항목이 여기서 해결되는 게 목표.
- 실패 → Tier 2.

### Tier 2 — 질의 확장 + 리랭크
- 질의를 여러 변형으로 펼치고(multi-query / HyDE: 가상 답변을 만들어 그걸로 검색), 1차로 많이 긁은 뒤 **리랭커(cross-encoder)**로 정밀 재정렬해 top-k를 추림.
- "1차 대충 많이 → 2차 정밀 추림"이 정확도 핵심 상승 구간.
- 실패 → Tier 3.

### Tier 3 — 에이전트형 반복 검색
- LLM이 "뭐가 부족한지" 보고 후속 질의를 스스로 만들어 여러 번 검색·종합. 여러 문서·여러 표에 흩어진 근거를 모아야 하는 어려운 항목용.
- 실패 → Tier 4.

### Tier 4 — 근거 없음 / 사람
- 자동으로 답을 지어내지 않는다. "해당 항목 근거 미발견"으로 명시하고, 시도한 search_terms·티어를 사유로 단다. 실사 응답에서 빈칸을 거짓으로 채우는 것보다 "미발견"이 훨씬 안전하다.

## 5. 게이트 B — 근거 게이트 (생성 *후*)

검색이 좋아도 LLM이 근거를 무시하거나 지어낼 수 있다. 생성된 답을 청크에 **대조**한다.

### 5.1 신호

- **G1. 인용 강제.** 답의 모든 주장·숫자에 출처 청크 ID가 달려 있는가? 인용 없는 문장이 있으면 fail.
- **G2. 숫자 출처 대조 (ESG 특화, OCR의 합계검증에 대응하는 강력한 게이트).** 답에 등장한 **모든 숫자**가 인용 청크 안에 글자 그대로 존재하는가? 청크에 없는 숫자는 환각(hallucination) — 즉시 fail. 싸고(문자열 매칭) 강력함.
- **G3. 함의 검증(entailment).** 인용 청크가 그 주장을 실제로 뒷받침하는가? NLI 모델이나 LLM 심판으로 "근거 → 주장"이 성립하는지 확인. (회색지대에서만)
- **G4. 단위·기간 일치.** 답의 단위·연도가 인용 청크의 그것과 같은가? (unit 필드, 기간 헤더 대조)
- **G5. 적절한 기권(abstention).** 검색 게이트가 실패했는데도 답이 단정적이면 fail — "근거 없음"으로 빠져야 함.

### 5.2 판정 정책

```
hard fail (하나라도 → 재처리):
  - G1 인용 없는 주장 존재
  - G2 청크에 없는 숫자 존재
  - G5 근거 없는데 단정

soft flag (각 1점): G3 약한 함의, G4 단위/기간 불일치

결정:
  - hard 0 AND soft ≤ 1  → ACCEPT (답변 확정 + 증빙 링크)
  - hard 0 AND soft 2+   → G3 호출(LLM 심판) → 통과 시 ACCEPT
  - hard ≥ 1             → 재생성 1회(같은 청크로 다시, "청크 밖 정보 쓰지 말 것" 강조)
                           재생성도 fail → 검색 티어 ESCALATE 또는 HUMAN
```

> 핵심 대응 관계: OCR의 **합계 검증**(숫자가 서로 맞나) ↔ RAG의 **숫자 출처 대조**(답의 숫자가 근거에 있나). 둘 다 싸고 결정적이고 강력한 ESG 특화 게이트다.

## 6. 추적 & 증빙 메타데이터

답의 각 값에 부착:
- `evidence_chunk_id` → `source_doc` → `source_page` / `bbox`
- `retrieval_tier` — 어느 티어에서 찾았는지(0~3)
- `retrieval_scores` — 검색 게이트 통과 시점 점수
- `grounding_status` — 근거 게이트 결과(숫자 대조 통과 여부 등)

이게 있어야 실사 양식의 각 답변에 "근거: 한울정밀 환경보고서 p.12, Tier 1 검색, 숫자 대조 통과"처럼 증빙을 1:1로 붙일 수 있다. OCR 문서의 메타데이터(bbox·page)와 그대로 이어진다 — OCR이 단 좌표를 RAG가 증빙 링크로 승계.

## 7. 평가 — "240 테스트 통과"와 다른 것

코드가 안 터지는 것(단위 테스트)과 검색이 정확한 것은 별개다. 필요한 것:

- **검색 평가셋.** 질의 N개 + 각 질의의 정답 청크(라벨). 측정: **Recall@k**(정답 청크를 top-k에 넣는 비율), **MRR**(정답 순위).
- **근거 평가셋.** 답 + "이 답이 근거에 충실한가" 라벨. 측정: faithfulness, 환각률, 적절 기권율.
- 이 평가셋이 게이트 임계치(0.6, 0.1, 20% 등)를 보정하는 근거가 된다. **없으면 이게 RAG 1순위 작업.** (OCR 평가셋 구축과 묶어서 진행)

## 8. 현업(개발자)에게 확인할 것

1. **지금 검색이 키워드냐, 임베딩이냐, 하이브리드냐?** (Tier 몇까지 이미 있는지)
2. **답변에 출처 청크가 강제로 달리는 구조냐, 사후에 붙이냐, 아예 없냐?** (G1 가능 여부)
3. **답의 숫자를 청크와 대조하는 검증이 있냐?** 없으면 G2가 가장 가성비 높은 첫 작업.
4. **리랭커가 파이프라인에 있냐?** 없으면 Tier 2가 큰 정확도 상승 여지.
5. **검색 평가셋(질의-정답청크 라벨)이 있냐?** "240 passed"가 검색 재현율을 재는 건지 확인.
6. **search_terms/unit 필드가 검색 질의에 실제로 주입되고 있냐, 스키마에만 있냐?**
7. **OCR이 남긴 page/bbox를 청크가 물려받아 저장하냐?** (6절 증빙 승계 가능 여부)

---

## 9. 구현 우선순위 (제안)

1. **PR-A: 근거 게이트 G1·G2 먼저.** 인용 강제 + 숫자 출처 대조. 검색을 안 건드려도 환각을 즉시 줄임 — 가성비 1위.
2. **PR-B: 검색 게이트 R1·R3.** 못 찾았을 때 지어내지 않게. 필드 커버리지(단위/핵심어)부터.
3. **PR-C: 평가셋 구축 + 임계치 보정.** (OCR 평가셋과 함께)
4. **PR-D: 하이브리드(Tier 1) + 리랭크(Tier 2).** 검색 재현율 본격 상승.
5. **PR-E: 에이전트형 Tier 3 + 증빙 메타데이터 승계.**

> 속도 최적화(임베딩 캐싱, top-k 축소, 쉬운 항목은 티어 스킵)는 정확도 안정화(PR-C) 이후 별도 트랙.
